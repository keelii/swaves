# Swaves 工程规范指南

本文件汇总了近期开发线程中沉淀出的工作规则，
用于后续变更时作为可执行的实践基准。

## 类别 0) 治理

### 0) 规则治理

- 这些规则默认强制生效；除非获得明确批准，否则不得违反。
- 若任务与这些规则冲突，执行前必须先与负责人确认。
- 在执行任何获批例外后，必须明确确认是否需要更新 `AGENTS.md`。

## 类别 A) 架构与路由

### 1) 核心原则

- 先保证行为正确，再优化简洁性与可维护性。
- 每个关注点（路由、路径、状态、配置）优先保持单一可信来源。
- 当有助于可读性时，允许小范围、刻意的重复。
- 对基础接线类任务（例如路由注册）避免过度抽象。
- 不要过度抽象和封装；代码应直观、符合直觉，不要为了形式堆叠多层抽象。
- 保持用户可见行为与系统真实能力一致。
- 若类型信息或静态分析已能证明安全，不要仅为风格添加额外防御性检查。
- 当 nil/empty 保护的是并发或生命周期边界（例如共享可变 resolver 指针、可选依赖、启动顺序）时，必须保留该保护。

### 2) URL 与路径语义

- 严格区分路由路径与内容 URL 前缀语义。
  - `ByPrefix` 风格值用于公开内容 URL 拼接。
  - 路由语义值用于 Fiber 路由注册与重定向。
- 不要把 prefix 语义混入路由构造逻辑。
- 内部链接与重定向使用命名路由和 `url_for`。
- 模板与 JS 中避免硬编码 dash 路径。

### 3) Dash 路径与重定向

- 任何跳转到 dash 页面的重定向都必须遵循可配置的 dash base path。
- 在可行场景下，用命名路由解析替代字符串拼接重定向。
- 引入新 dash 页面时，必须分配路由名称。

### 13) 模板引擎与上下文契约

- 模板运行时仅使用 MiniJinja；禁止引入双引擎切换或回退路径。
- 站点与管理后台模板文件统一使用 `.html`。
- 模板上下文保持扁平且显式；不得恢复 `Req/Auth/Site/__root` 包装命名空间。
- 业务绑定不得覆盖渲染层已注入字段（例如 `RouteName`、`Query`、`IsLogin`、`_csrf_token_value`）。
- 避免将请求 locals 盲目全量注入模板；通过渲染辅助函数显式传入稳定字段。
- 模板中的 `import/include` 路径使用模板根相对路径，并显式带 `.html` 后缀（不要求 `/` 前缀）。
- `include` 场景尽量不要使用 `with` 传参；在可行前提下，由被 include 模板直接读取当前上下文所需字段。
- 模板中的内部链接必须继续使用 `url_for`，不得硬编码 dash 路径。
- 模板热重载由 `SWAVES_ENV` 推导：非 `prod` 可按请求清缓存，`prod` 禁止按请求清缓存。

## 类别 B) 产品流程与数据语义

### 4) 资源库规则

- 运行时仅使用一个资源提供商。
- 资源上传/列表页面不展示提供商选择 UI。
- 提供商切换仅通过设置中的默认配置进行。
- 资源列表页读取数据库记录，而非调用 provider 列表 API。

### 5) 设置与前缀字段

- `PrefixValue` 表示前缀来源/值，不是展示说明文本。
- 前端中前缀字段的联动仅是 UI 行为。
- UI 联动不得隐式改变存储语义。
- 校验规则必须与当前字段含义及 URL 组合行为保持一致。

### 6) 导入流程

- 使用异步导入，并具备显式状态（例如 `importing`）。
- 导入项在最终确认前应可编辑。
- 确认动作应完成最终落库并写入编辑后的值。
- 取消动作必须清理关联记录（posts、tags、categories links）。
- 向用户返回具体失败原因，而不是仅返回通用 “import failed”。

## 类别 C) 运行时可靠性与安全性

### 7) 错误处理与日志

- 任何已处理的错误路径都应包含带上下文的显式日志。
- 面向用户的 API 错误应包含可执行、具体的信息。
- 外部 provider 失败时应包含状态码与简要响应细节。
- 不要在 handler 或后台任务中静默吞掉错误。

### 8) 数据库初始化与迁移

- 不要把迁移逻辑放入 `InitDatabase`。
- 初始化流程中仅保留 `EnsureDefaultSettings`。
- 任何 schema/data migration 必须在 `InitDatabase` 之外实现。

### 10) Fiber v3 升级指南

- 升级改动前先核对 v2 与 v3 的 API 迁移点。
- 示例：`DisableStartupMessage` 在 v3 属于 `fiber.ListenConfig`。
- 升级后验证 session 编解码兼容性。
- 中间件依赖升级时保持主版本匹配。

### 14) 后台任务生命周期

- 任务注册表初始化必须具备幂等性和并发安全。
- 长生命周期调度器初始化应在应用 listen hook 启动，而不是基础 app 构建阶段。
- 关闭流程要对称：停止调度器/cron 资源并等待优雅停止完成。
- 注册表与执行器代码必须记录明确的跳过/失败原因（例如 nil store、重复初始化、未注册任务 code）。

## 类别 D) 代码组织与配置卫生

### 9) 模型、处理器与复用

- 数据库操作尽可能放在 models 层。
- handlers 负责流程编排，不要内嵌复杂 SQL 逻辑。
- 可复用 JS 工具统一沉淀到 `web/static/dash/main.js` 及站点对应入口。
- 共享 JS 工具优先使用原生 DOM API，不应依赖额外库。

### 11) 环境变量与配置卫生

- 环境变量定义统一收敛在 `env.go`。
- 使用一致的前缀与命名规范。
- 及时清理未使用的环境变量。
- 当直接 `os.Getenv` 足够时，避免为 env 名称再新增重复常量。

## 类别 E) 质量与交付

### 12) 测试策略

- 为纯 URL/路径组合逻辑补充单元测试。
- 删除已移除功能对应的过时测试。
- 跨模块重构后运行全量测试（`go test ./...`）。
- 除非明确延期项，所有回归必须修复。

### 15) PR 自检清单

合并前请核对以下项：

- [ ] 在可用命名路由处不再存在硬编码 dash URL。
- [ ] 新增路由均有稳定路由名称。
- [ ] 重定向与链接均遵循可配置 dash base path。
- [ ] Prefix/联动变更未意外改变持久化语义。
- [ ] 导入失败响应包含具体错误细节。
- [ ] 错误分支有包含上下文的日志。
- [ ] `InitDatabase` 中未新增迁移逻辑（`EnsureDefaultSettings` 除外）。
- [ ] 重复模板/JS HTML 块已同步或已合并统一。
- [ ] 模板绑定保持扁平显式上下文，未恢复 `Req/Auth/Site/__root` 包装层。
- [ ] 模板 `import/include` 路径使用模板根相对 `.html` 路径（不依赖 `/` 前缀）。
- [ ] 模板重载行为符合 `SWAVES_ENV` 推导（非 `prod` 可按请求清缓存）。
- [ ] Job registry 生命周期保持安全（listen hook 初始化 + shutdown 销毁）。
- [ ] 新增/更新测试覆盖新行为并移除遗留行为测试。
- [ ] `go test ./...` 全量通过。
