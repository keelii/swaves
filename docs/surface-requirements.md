# Swaves 三端功能需求

## 1. 产品定义

- Swaves 的 Web 形态分为两个入口：
  - Web 公开端：面向访客的内容展示与互动入口。
  - Web 管理端：面向管理员的浏览器后台入口。
- Swaves 的 Native 形态只有一个入口：
  - Native 管理端：面向高频作者与运营的桌面后台入口。
- 内容、设置、任务、通知、资源元数据等系统状态以服务端为唯一真相源。
- Web 管理端与 Native 管理端共享同一套管理域能力，差异只体现在终端交互与本地能力。

## 2. 统一领域能力

### 2.1 内容生产

- 文章与页面的创建、编辑、预览、保存、发布、撤回、删除、恢复。
- Markdown 内容编辑与辅助能力。
- 加密文章的创建、编辑、删除与过期清理。

### 2.2 内容结构

- 标签管理。
- 分类管理与树形层级调整。
- 重定向管理。
- 站点 URL 规则与前缀策略的配置与使用。

### 2.3 用户互动

- 公开端点赞与 UV 统计。
- 评论提交、去重、频控与审核。
- 管理端通知中心与未读状态。

### 2.4 资源与数据流转

- 资源库管理。
- 文件上传与删除。
- Markdown 导入工作流：解析、编辑、确认、取消清理。
- 数据导出与备份。

### 2.5 系统运维

- 站点设置与后台设置管理。
- 定时任务、立即触发、运行记录查看。
- 监控、指标、错误日志查看。
- 回收站与硬删除。

## 3. Web 公开端需求

### 3.1 目标

- 面向访客提供稳定、清晰、可分享的内容消费入口。
- 承担公开内容展示、分发和轻互动，不承担管理职责。

### 3.2 核心功能

- 首页与文章列表浏览。
- 文章详情页访问。
- 页面详情访问。
- 分类页与标签页访问。
- 404 页与错误页展示。
- RSS 输出。
- 原始 Markdown 内容访问。
- 点赞操作。
- 评论提交与结果反馈。
- 访客标识与基础互动状态记忆。

### 3.3 体验要求

- URL 稳定，路由语义与站点配置一致。
- 页面渲染对搜索引擎与分享场景友好。
- 内容阅读体验优先，移动端可用。
- 错误路径可恢复，返回路径明确。
- 公开端的交互能力必须诚实反映真实状态，不能出现“看起来成功但服务端未成功”的假反馈。

### 3.4 不承担的职责

- 不提供文章编辑、资源管理、导入导出、系统设置、任务运维等后台操作。
- 不依赖桌面端能力完成核心访问流程。

## 4. Web 管理端需求

### 4.1 目标

- 作为完整、可远程访问的后台入口。
- 在任何有浏览器的环境中，都能完成站点的全部管理工作。
- 作为 Native 管理端之外的标准管理基线与兜底入口。

### 4.2 核心功能

- 登录、登出、会话维持与 CSRF 防护。
- 工作台首页与关键指标总览。
- 文章与页面的列表、筛选、创建、编辑、发布、删除与恢复。
- 记录视图与内容状态查看。
- 评论审核与批量处理。
- 通知列表、未读数、单条/全部已读、删除。
- 标签管理。
- 分类管理与树形调整。
- 重定向管理。
- 资源库浏览、上传、删除、批量删除。
- 加密文章管理。
- 设置列表、全量设置、设置项新增与编辑。
- 回收站查看、恢复、彻底删除。
- HTTP 错误日志查看与清理。
- 定时任务管理、手动触发、运行记录查看。
- Markdown 导入工作台与导出能力。
- 监控页、指标 API、监控数据查询。

### 4.3 体验要求

- 浏览器即可完成全部后台能力，不要求用户安装桌面应用。
- 功能完整优先于终端增强，是管理域的标准入口。
- 所有批量操作、破坏性操作和长流程操作都需要明确反馈。
- 需要支持远程访问与跨设备使用。
- 不允许把关键流程做成只能靠 Native 能力完成。

### 4.4 不承担的职责

- 不承担公开站点的阅读体验优化职责。
- 不以本地文件系统或本地应用状态作为系统真相来源。

## 5. Native 管理端需求

### 5.1 目标

- 面向高频创作、批量操作和系统运维，提供比浏览器后台更高效的桌面体验。
- Native 管理端只做管理，不承载公开内容展示职责。
- Native 管理端不是另一套业务系统，而是管理域的桌面入口。

### 5.2 共享管理能力覆盖

- Native 管理端需要覆盖与 Web 管理端一致的核心管理域能力。
- 内容生产、内容结构、资源流转、设置、任务、通知、回收站等模块在语义上必须与 Web 管理端一致。
- 同一操作在 Web 管理端与 Native 管理端上应得到相同的业务结果、校验规则和状态定义。

### 5.3 Native 专属增强能力

- 站点连接管理：保存一个或多个站点连接配置。
- 本地安全凭证存储。
- 原生文件打开与保存对话框。
- 拖拽导入 Markdown 文件与拖拽上传资源。
- 剪贴板图片或文件粘贴上传。
- 长文编辑的自动保存、草稿缓存与崩溃恢复。
- 导入、上传、导出、任务执行等长流程的原生进度反馈。
- 系统通知。
- 单实例运行与窗口状态持久化。
- 键盘快捷键、命令面板、最近操作入口等桌面效率能力。
- 网络异常提示、失败重试与任务恢复提示。

### 5.4 体验要求

- 优先服务高频作者和运营，而不是偶发访问者。
- 编辑、导入、上传、设置这类高频场景应比浏览器后台更快、更顺手。
- 原生能力只做增强，不改变管理域的业务真相。
- 桌面端必须清楚地区分“本地状态”和“服务端状态”。

### 5.5 不承担的职责

- 不承担公开文章、分类、标签、RSS 的消费入口职责。
- 不承担 SEO、分享落地页、公开评论入口等公开站点能力。
- 不直接以本地数据库文件替代服务端作为内容真相源。

## 6. 管理端共享要求

- Web 管理端与 Native 管理端必须共享同一套管理域模型。
- 相同实体必须使用相同字段语义、状态语义与校验规则。
- 相同流程必须使用相同的成功条件、失败条件与错误消息结构。
- 搜索、筛选、分页、批量操作的行为定义必须保持一致。
- 通知、任务、运行记录、导入导出、回收站等状态模型必须保持一致。
- 管理域业务逻辑不能在 Web 管理端与 Native 管理端各自分叉实现。

## 7. 实现优先级

### 7.1 P0

- Web 公开端：内容访问、RSS、点赞、评论、错误页。
- Web 管理端：完整后台可用，覆盖所有现有管理模块。
- Native 管理端：文章/页面编辑、资源管理、导入导出、设置、任务、通知。

### 7.2 P1

- Native 管理端补齐评论、标签、分类、重定向、回收站等中频模块。
- 管理双端在工作流和反馈上进一步对齐。

### 7.3 P2

- Native 管理端补齐监控、错误日志、加密文章等低频但专业化模块。
- Native 管理端增强站点连接管理、快捷操作和长流程恢复能力。
